# TraderClaw Install Contract (v1.0.7)

This document defines the canonical installation and onboarding contract for `solana-traderclaw-v1@1.0.6` (npm package id aligned with OpenClaw manifest `solana-traderclaw-v1`; legacy `traderclaw-v1@1.0.7` installs may show an id-hint warning until migrated).

Use this as the source of truth for:
- CLI docs
- external installer docs
- GUI installer behavior

---

## 1) Goals

A successful install means all of the following are true:
- plugin is installed and loaded as `solana-traderclaw-v1`
- setup completed with wallet + session credentials written
- user can choose either:
  - **Quick local mode** (no gateway registration yet), or
  - **Event-driven mode** (gateway credentials registered and active)
- user has copy-paste verification commands

---

## 2) Inputs

Required:
- OpenClaw runtime installed (`openclaw`)
- Node.js >= 22
- Orchestrator URL (default: `https://api.traderclaw.ai`)

Optional:
- API key (`oc_*`)
- Gateway URL/token for event-driven callbacks

Derived:
- wallet keys/session tokens generated by setup flow

---

## 3) Setup Lanes

### Lane A: Quick Local (default beginner path)

Purpose:
- complete setup with minimal friction
- postpone gateway exposure work

Command shape:

```bash
traderclaw setup --signup --url https://api.traderclaw.ai --skip-gateway-registration
```

Expected result:
- setup succeeds
- trading tools available
- event-driven wakeups not yet enabled

### Lane B: Event-Driven (advanced/recommended for production)

Purpose:
- enable orchestrator push callbacks and wakeups

Command shape:

```bash
traderclaw setup --signup --url https://api.traderclaw.ai \
  --gateway-base-url <gatewayBaseUrl> \
  --gateway-token <gatewayToken>
```

Expected result:
- setup succeeds
- gateway credentials become active
- alpha/event forwarding can wake the agent

---

## 4) Gateway Rules

- `gatewayBaseUrl` must be reachable from orchestrator network perspective.
- `localhost` / `127.0.0.1` are invalid for remote callback registration.
- For tailnet/private baseline, use:
  - `http://<tailscale-ip>:18789`
- For HTTPS serve mode, use:
  - `https://<tailscale-hostname>`

---

## 5) Decision Points (Installer UX)

Installers must present an explicit lane choice:
- **Quick local now**
- **Enable event-driven now**

If event-driven is chosen but registration fails:
- offer retry
- allow fallback to quick local completion

Installers must never dead-end users without a fallback path.

---

## 6) Verification Outputs

### Minimum success checks

1) Plugin loaded:

```bash
openclaw plugins list
```

2) Session/wallet health:

```bash
traderclaw status
```

3) Gateway health (event-driven lane):

```bash
curl -H "Authorization: Bearer <gatewayToken>" <gatewayBaseUrl>/health
```

4) Credentials active (event-driven lane):
- via `solana_gateway_credentials_get()`, expect `active: true`

---

## 7) Standardized Post-Install Summary

Every installer should print this outcome block:

- Mode: `Quick Local` or `Event-Driven`
- Plugin: installed version
- Session: `ACTIVE` or clear remediation
- Wallet: `ACTIVE` and public address
- Gateway credentials: `SKIPPED` or `ACTIVE`
- Next 3 commands to continue

---

## 8) Failure Handling Contract

Common failures and expected recovery:

- Session expired:
  - `traderclaw login`
- Gateway not reachable:
  - verify bind/exposure mode and token
- No funds:
  - show exact wallet address and minimum suggested SOL

Non-blocking warnings:
- plugin id mismatch warning
- uuid schema warning

---

## 9) Security Baseline

- Never log secrets in plaintext:
  - API keys
  - refresh tokens
  - wallet private keys
  - gateway bearer tokens
- Mask secrets in UI and logs.
- Prefer HTTPS/WSS where available; if using tailnet HTTP, clearly mark trust boundary assumptions.
